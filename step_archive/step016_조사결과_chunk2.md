# Step 016 조사결과 chunk2 - 대중 앱 사례

## 1. 카카오톡/메신저: 메시지 무결성 검증

- 파일 전송 시 송신자가 SHA-256 해시 계산 → 메타데이터에 첨부
- 수신자가 받은 파일을 다시 해시 → 일치하면 변조 없음 확인
- 직관: "내가 보낸 사진이 중간에 바뀌지 않았어요" 영수증

## 2. 비밀번호 저장: 절대 평문 저장 금지

- 회원가입: 비밀번호 → bcrypt/Argon2(salt 포함) → DB에 해시만 저장
- 로그인: 입력 비밀번호 해시 vs DB 해시 비교
- 직관: "비번을 회사도 몰라야 안전하다"
- 참고: bcrypt는 work factor(연산 비용)를 조정해 무차별 공격 방어

## 3. Git: 커밋 ID

- 모든 커밋·트리·블롭은 SHA-1(최근 SHA-256) 해시로 식별
- 파일 1바이트만 바뀌어도 커밋 해시가 완전히 달라짐 → 변조 즉시 감지
- 직관: "Git 커밋 해시 = 코드 스냅샷의 지문"

## 4. 비트코인/블록체인

- 각 블록 헤더 = (이전 블록 해시 + 트랜잭션 머클루트 + nonce) → SHA-256 두 번
- 작업증명: 결과 해시가 특정 0개수 이상으로 시작할 때까지 nonce 변경 (마이닝)
- 직관: "수많은 자물쇠를 시도해 정답 자물쇠를 찾는 게임"

## 5. IPFS / 콘텐츠 주소

- 파일 자체의 해시(CID)를 주소로 사용 → 같은 파일은 어디 있든 같은 주소
- HTTP는 위치 기반(URL), IPFS는 내용 기반(CID)
- 직관: "DNA가 같으면 어디서 발견되든 같은 사람"

## 6. CDN/HTTP 캐시: ETag

- 응답 헤더 `ETag: "W/\"abc123\""` 는 콘텐츠 해시
- 브라우저가 If-None-Match로 ETag 보내면 서버는 변경 없을 시 304 반환
- 직관: "이미 본 영상 재다운로드 안 함"

## 7. HashMap / Dict: 사전 검색의 핵심

- 자바스크립트 객체, Python dict, Java HashMap 모두 내부적으로 해시 테이블
- 키를 해시 → 버킷 인덱스 → O(1) 조회
- 직관: "사전 단어를 페이지 번호로 직행"

## 8. Bloom Filter: 스팸/캐시 적중 탐지

- 해시 함수 k개로 비트 벡터에 표시
- "있을 수도 없다 / 없을 가능성 매우 큼" 판정
- 직관: "확률 출입증" — Cloudflare/Chrome Safe Browsing이 사용

## 9. CDN/Subresource Integrity (SRI)

- `<script src="x.js" integrity="sha384-...">` 로 위변조 방지
- 외부 CDN이 해킹돼도 해시 불일치 시 브라우저가 거부

## 10. 토렌트/마그넷 링크

- 마그넷 URI에 info hash가 포함 → 같은 콘텐츠 식별자
- 피어 간 파일 검증도 청크 해시로 수행
